## 一、全链路运行流程

### 1.1 每日全流程拓扑（无变化）

```
main.py full_cycle ─┬─ Phase 1: Orchestrator ── 五池状态读取
                    ├─ Phase 2: News ────────── 新闻抓取+LLM分析 → S级驱动
                    ├─ Phase 3: Screen ──────── LLM快筛 → 候选池
                    ├─ Phase 4: Review ──────── LLM审查+过热检测 → 升级/降级
                    ├─ Phase 5: Skeptic ─────── 二审制Gate → 高风险拦截
                    ├─ Phase 6: Decision ────── LLM决策 → S池写入+兜底引擎
                    ├─ Phase 7: Market ──────── 行情刷新
                    ├─ Phase 8: Pool Refresh ── 四池价格刷新+评分衰减+降级扫描
                    └─ Phase 9: T+1 Tracker ── 闭环追踪+S池回流评估
```

```
快筛候选池 ──(LLM评分≥75)──→ 重点观察池 ──(决策主推)──→ S级操作池
     ↑                            │                      │

     │                            ↓         ┌───────────┘
     │         ┌─── 重点观察池 ←── 等比衰减 ├── T+1好→重点池
     │         │                           └── T+1差→边缘池 ←─── P1-4修
     │         │
     │   持仓池 ←──(买入执行)──┘
     │     │

     └──────────────────────┘
```

## 三、剩余问题评测（5项待修）

### 🔴 P1-5 过热检测未覆盖盘中价格刷新

**位置**：`agents/pool_manager.py` 4个 `refresh_*` 函数

**问题**：`OverheatDetector.detect()` 只在审查阶段和S池准入时被调用。盘中价格刷新时（每天2次+手动调用），4个refresh函数均不触发过热检测：

```
refresh_key_watch_prices()       ❌ 不过热
refresh_holdings_prices()        ❌ 不过热  
refresh_screen_candidate_prices()❌ 不过热
refresh_s_operation_prices()     ❌ 不过热
```

**风险场景**：某标的在审查阶段评分80分（不过热），下午盘中突然拉升+15%。`refresh_key_watch_prices` 在收盘前执行 → 价格更新了但不过热检测 → 评分保持80分 → 第二天继续被推荐买入 → 用户高位接盘。

**修复方案**：在4个refresh函数的存量循环中，对涨幅>5%的标的加 `OverheatDetector.detect()` 调用，检测到过热则扣分+标记。改动量：~20行。

---

### 🔴 P1-6 硬编码阈值残留（5处）

**位置**：多处文件，修后仍有残留

| 位置 | 硬编码值 | 问题 |
|------|----------|------|
| `track_recorder.py:65` | `>= 70` | 应为≥75（与S_POOL_MIN_SCORE对齐） |
| `pool_manager.py:1023` | `>= 70` A级 | 应为≥75（review_agent用75为A级） |
| `pool_manager.py:1262` | `>= 80` 优秀 | 应为≥75（与S_POOL_MIN_SCORE对齐） |
| `review_agent.py:910/964` | `>= 75` A级 | pool_manager用70为A级 → 两处不一致 |
| `decision_agent.py:1409` | `>= 70` 胜, `>= 60` 平 | 展示用，影响较小 |

**风险场景**：pool_manager定义70分为A级，review_agent定义75分为A级，导致用户看到同一标的在两个报告中等级不同（A级 vs B级）。

**修复方案**：统一为 `thresholds.py` 定义 `SCORE_A_LEVEL=75`、`SCORE_B_LEVEL=65`，所有文件引用常量。改动量：~10处。

### 🟡 P1-8 市场通缩兜底检查无安全余量

**位置**：`agents/decision_agent.py:593`

**问题**：兜底引擎检查固定使用 `>= 75`，但 `DYNAMIC_SCORE_THRESHOLDS` 定义偏空市场准入85分、震荡78分。固定75分在偏空市场中过低。

```python
# decision_agent.py L593 — 固定75分，不随市场状态调整
if s.get("score", 0) >= 75 and s.get("passed", False)
```

**风险场景**：市场偏空时（DYNAMIC阈值85分），一个78分的标的被LLM拒绝（偏空市场LLM正确选择空仓），兜底引擎启动 → 检查到78 ≥ 75 → 生成模板买入方案 → 偏空市场中大概率亏损。

**修复方案**：`decision_agent.py` L593 改为从 `self._market_state` 读取动态阈值：`min_score = self._market_state.get("dynamic_entry", 75)`。改动量：3行。

### 🟡 路径绕过：scripts/pool_price_refresh.py

**位置**：`scripts/pool_price_refresh.py:76-117`

**问题**：手动实现了S池行情刷新逻辑（读取→fetch行情→写入），不调用 `pool_manager.refresh_s_operation_prices()`，绕过降级扫描和评分衰减。

```python
# L76-117：手动实现，非pool_manager方法
s_pool_file = pool_dir / "S级操作池.json"
... fetch行情 ...  # 无降级扫描、无评分衰减
```

**风险场景**：运维人员手动执行 `pool_price_refresh.py` 刷新行情 → S池价格更新但降级扫描不执行 → 评分<65的标的滞留在S池 → 过期后T+1清理流程可能因评分错误导致错误回流。

**修复方案**：`scripts/pool_price_refresh.py` L76-117 替换为 `pm.refresh_s_operation_prices()` 调用。改动量：~3行。
