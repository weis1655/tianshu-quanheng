## 二、模式匹配分析

### 2.1 关键发现：`delegate_task` 不存在

> **重要**：背景中描述的"每步用 delegate_task 实现子代理协作"已过时。实际代码中：
> - **无 `delegate_task` 函数/方法**
> - **无 `TradingAgent` 类**（建仓由 `PoolManager.add_to_holding()` 直接完成）
> - 所有 Agent 通过 **主进程串行调用**（`main.py` → `run_phase()`）
> - Agent 间通过 **文件系统** 通信（写 `.md` / `.json`，下一个 Agent 读）

### 2.2 逐组件匹配表

| 天枢组件 | 当前模式 | Harness匹配模式 | 优缺点 |
|----------|----------|----------------|--------|
| **Orchestrator** | 规则路由（时间+用户意图→阶段选择） | **Supervisor** (弱匹配) | ✅ 0次LLM，纯规则，轻量可靠<br>❌ 无动态分发、无负载感知、无并行决策 |
| **main.py 串联** | 串行函数调用（A→B→C→D） | **Pipeline** | ✅ 流程清晰，失败级联终止<br>❌ 无法并行，总耗时=各步LLM耗时之和 |
| **NewsAgent** | 输入→LLM→输出报告 | **Pipeline** (Stage 1) | ✅ 职责单一<br>❌ 与其他Agent无结构耦合 |
| **ScreenAgent** | 新闻驱动+规则→LLM推荐→池更新 | **Pipeline** (Stage 2) | ✅ 双盲设计（不看五池）<br>❌ 上下文注入过多（行情+五池+大盘） |
| **ReviewAgent** | 四维评分+LLM→流转方向 | **Producer-Reviewer** 的半边 + **Expert Pool** (弱) | ✅ 结构化输出（四维表格）<br>❌ "双盲"被破坏（ReviewAgent参考快筛驱动级别）<br>❌ 同时做评分+池更新，职责过重 |
| **SkepticAgent** | JSON格式五维质疑+Gate控制 | **Producer-Reviewer** ( reviewer 角色) | ✅ 纯质疑不推荐，角色清晰<br>✅ 精确的Gate逻辑（代码层，非LLM层）<br>❌ Gate逻辑散落在DecisionAgent中，核心门控与消费方耦合 |
| **DecisionAgent** | 多数据注入→LLM→结构化解析 | **Pipeline** + **Hierarchical Delegation** (弱) | ✅ 策略补丁丰富（空仓兜底/二次尝试/模板化）<br>❌ **职责过载**：池管理(S级过期)+Gate过滤+行情检查+LLM决策+池更新+回溯记录 |
| **PoolManager** | JSON文件读写+容量限制+排序 | **Fan-out/Fan-in** (池查询场景) | ✅ 统一的CRUD接口<br>❌ 每个Agent自行实例化PoolManager，存在多个实例 |
| **MarketAgent** | 0 LLM API行情获取 | 基础设施（不匹配 Harness 模式） | ✅ 纯工具，可靠<br>❌ 与决策流程耦合（行情缺失阻断决策） |
| **ReviewEvo** | JSON日志+权重计算 | **Expert Pool** (弱) 知识库 | ✅ 记录决策历史<br>❌ 权重自动调整未看到实际使用证据 |
| **FeedbackLoop** | 持仓分析+胜率+熔断检查 | **Expert Pool** (监控专家) | ✅ 熔断检查有实际价值<br>❌ 只读不写，与决策流程解耦 |
| **MemPalace** | 三元组知识图谱 | **Expert Pool** (记忆专家) | ✅ 跨日记忆保留<br>❌ 保存后无实际读取利用 |
| **weekly_review_agent** | 池卫生+假设验证+权重修正 | **Hierarchical Delegation** (弱) | ✅ 周期性独立任务<br>❌ 通过subprocess调用，非Agent协作 |

## 三、架构升级建议

### P0: 职责分离 — 将 DecisionAgent 拆分为多Agent协作

- **现状问题**：
  - `decision_agent.py` 高达 **1413 行**，是第二大文件（仅次于 pool_manager.py）
  - 同时负责：S级过期清理、Gate过滤、行情检查、LLM决策、池更新、回溯记录、闭环追踪
  - 违反**单一职责原则**——一个类做了6件事
  - 策略"补丁"堆积（空仓兜底、二次尝试、模板化方案），核心逻辑和异常处理交织

- **Harness方案**：
  ```
  ┌────────────────────────────────────────────────────┐
  │  拆分为 Harness Producer-Reviewer + Supervisor      │
  │                                                    │
  │  Supervisor(DecisionSupervisor)                     │
  │   ├─ PlanProducer(原决策主体) → LLM生成方案          │
  │   ├─ RiskReviewer(原Gate逻辑) → 合规检查+模板兜底    │
  │   ├─ PoolUpdater(池操作) → 纯CRUD                   │
  │   └─ TrackRecorder(闭环) → 数据记录                  │
  └────────────────────────────────────────────────────┘
  ```

- **预期收益**：
  - 每段代码行数降至300-500行，可维护性提升
  - 单元测试可独立编写
  - 策略补丁可按需替换（如模板兜底可升级为独立Agent）

- **改造成本**：
  - 代码量：约 600-800 行（拆分+接口对接）
  - 复杂度：中等，主要是文件拆分和依赖整理

- **风险**：
  - 低风险。逻辑不变，只是代码重组。

---

### P1: 引入 Fan-out/Fan-in 批量评估模式

- **现状问题**：
  - `ReviewAgent._batch_enrich_key_watch()` 方法（行350-495）已经**内嵌了Fan-out模式**的雏形，但实现方式是：
    - 在ReviewAgent内部直接调 `PoolManager._call_llm_for_limits()` — 违反职责边界
    - 与 ReviewAgent 的"审查"职责无关，属于建仓前评估
  - 各Agent重复拉行情（ScreenAgent拉→ReviewAgent拉→SkepticAgent拉→DecisionAgent拉），每次调用 `fetch_quotes` 都有网络延迟

- **Harness方案**：
  ```
  ┌───────────────────────────────────────────────────┐
  │  Fan-out: 同时评估重点观察池所有标的               │
  │                                                    │
  │  ┌─────────┐  ┌─────────┐  ┌───────────┐          │
  │  │买入价评估│  │止损评估  │  │止盈评估    │          │
  │  └────┬────┘  └────┬────┘  └─────┬─────┘          │
  │       └────────────┼──────────────┘                │
  │                    ▼                               │
  │           ┌────────────────┐                       │
  │           │  Fan-in: 汇总   │                       │
  │           │  输出结构化数组  │                       │
  │           └────────┬───────┘                       │
  │                    ▼                               │
  │           ┌────────────────┐                       │
  │           │  PoolUpdater    │                       │
  │           │  批量写入重点池  │                       │
  └───────────────────────────────────────────────────┘
  ```

  - **行情一次性拉取**：引入共享 `QuoteService` 或 `MarketCache`，所有Agent共用一次行情查询结果
  - **Lazy loading**：只在需要时才拉，但结果缓存到内存+文件（30秒过期）

- **预期收益**：
  - 减少50%-70% 重复API调用
  - LLM调用从 N次（每只股票）→ 1次（批量），N=5-20只，节省4-19次LLM调用
  - 代码内聚性提升

- **改造成本**：
  - 代码量：约 300 行（QuoteService + 批量评估重构）
  - 复杂度：中等

- **风险**：
  - 批量LLM调用可能受 token 上限限制，需要智能分块
  - 缓存过期策略需谨慎，避免使用过期行情

### P2: Gate 逻辑独立化 — SkepticAgent 门控职责回归

- **现状问题**：
  - `SkepticAgent` 输出 JSON 裁决文件（`质疑审查裁决.json`）
  - `DecisionAgent` 读这个文件来执行门控逻辑（`blocked_codes` 过滤、连续3次阻塞自动降级）
  - **门控逻辑和决策逻辑交织在同一个文件中**
  - 门控的核心状态（`blocked_count`）写在重点观察池 JSON 中，与池数据耦合

- **Harness方案**：
  ```
  ┌──────────────────────────────────────────────────┐
  │  独立Gate服务（Producer-Reviewer Gate角色）        │
  │                                                    │
  │  SkepticAgent 输出 JSON 裁决                       │
  │       │                                            │
  │       ▼                                            │
  │  GateController（新模块，0次LLM）                   │
  │    ├─ 读取裁决 → 生成 block_list                   │
  │    ├─ 更新 blocked_count（独立状态文件）            │
  │    ├─ 判断≥3次 → 直接写入边缘池（跳过DecisionAgent）│
  │    └─ 输出 filtered_pools（已排除阻塞标的）         │
  │       │                                            │
  │       ▼                                            │
  │  DecisionAgent（只接收过滤后的pools）               │
  └──────────────────────────────────────────────────┘
  ```

- **预期收益**：
  - DecisionAgent 减少约 150 行 Gate 逻辑
  - Gate 状态不污染池 JSON
  - 可独立测试 Gate 逻辑（3次阻塞→降级的业务规则）

- **改造成本**：
  - 代码量：约 200 行（GateController新文件）
  - 复杂度：低

- **风险**：
  - 需要修改 DecisionAgent 和 main.py 的调用链
  - 低风险

### P3: 引入 Expert Pool 统一记忆和查询

- **现状问题**：
  - 记忆散落在多处：
    - `ReviewEvo`（决策日志 + 权重）
    - `ClosedLoopTracker`（闭环追踪 JSON）
    - `MemPalace`（知识图谱三元组）
    - `feedback_loop.py`（持仓分析 + 胜率）
  - 彼此不互通，信息孤岛
  - 权重自动调整只有框架没有实际使用

- **Harness方案**：
  ```
  ┌──────────────────────────────────────────────────────┐
  │  Expert Pool: 记忆专家团队                              │
  │                                                        │
  │  ┌─────────────────┐                                  │
  │  │ MemorySupervisor  │ ← 统一调度入口                     │
  │  └────────┬─────────┘                                  │
  │      ┌────┼────┬────┬────┐                             │
  │      ▼    ▼    ▼    ▼    ▼                             │
  │  HxExpert StatsExpert WeightExpert GateExpert EvoExpert│
  │  (历史) (统计)  (权重)  (门控) (复盘)                   │
  └──────────────────────────────────────────────────────┘
  ```

- **预期收益**：
  - 清晰的记忆读取路径（`screen_agent` 需要历史权重时，问 WeightExpert）
  - 可垂直扩展（增加新的 Expert 不影响现有流程）

- **改造成本**：
  - 代码量：约 400 行（Supervisor + Expert 基类 + 实现）
  - 复杂度：中高（需要设计统一接口）

- **风险**：
  - 过度工程风险（当前记忆查询需求有限）
  - **建议降级为 P2 或只做统一接口不做Supervisor**

### P4: Supervisor 模式 — 升级 Orchestrator 为动态调度器

- **现状问题**：
  - `Orchestrator` 只有 138 行，功能有限
  - 不支持动态任务分发
  - 不支持并行执行（市场数据+持仓刷新当前串行）
  - 不支持条件分支（如"如果审查报告截断→降级+补充审查"）

- **Harness方案**：
  ```
  ┌──────────────────────────────────────────────────┐
  │  Dynamic Supervisor                               │
  │                                                    │
  │  main.py → Supervisor.run()                        │
  │    ├─ Stage: news (串行)                           │
  │    ├─ Stage: market + pool_refresh (并行!)         │
  │    ├─ Stage: screen (串行)                         │
  │    ├─ Stage: review (串行)                         │
  │    │   └─ 条件分支: truncated? → re-review ✗      │
  │    ├─ Stage: skeptic (串行)                        │
  │    │   └─ 条件分支: all_blocked? → skip_decision   │
  │    └─ Stage: decision (串行)                       │
  └──────────────────────────────────────────────────┘
  ```

- **预期收益**：
  - market + pool_refresh 可并行（节约 2-5 秒）
  - 条件分支让流程更灵活
  - 是后续功能扩展的基础设施

- **改造成本**：
  - 代码量：约 400 行
  - 复杂度：中（需要任务调度框架或简单线程池）

- **风险**：
  - 低。并行化需要考虑 Python GIL 问题，但 IO 密集型任务（行情API）不受影响
  - 非必需——当前串行对总耗时影响不大（LLM调用本身是主要瓶颈）

## 四、总结

### 4.1 最适合天枢的 Harness 模式组合

```
主力模式：           Pipeline（串行依赖任务流）
                      ├─ NewsAgent → ScreenAgent → ReviewAgent
                      │  → SkepticAgent → DecisionAgent
                      └─ 级联终止机制

辅助模式1：          Producer-Reviewer（产出+审查）
                      ├─ ScreenAgent (Producer)
                      ├─ ReviewAgent (1st Reviewer)
                      ├─ SkepticAgent (2nd Reviewer/Gate)
                      └─ 二审制Gate

辅助模式2：          Fan-out/Fan-in（批量并行）
                      └─ 重点观察池批量评估（替代逐只LLM调用）

辅助模式3：          Expert Pool（知识库）
                      ├─ ReviewEvo + ClosedLoop + MemPalace
                      └─ 统一记忆访问

推荐不采用的模式：    Hierarchical Delegation
                      └─ 天枢深度有限（2层：main.py→Agent），不需要递归分解
```

### 4.2 建议实施方案（分三阶段）

| 阶段 | 内容 | 工期估计 | 收益 |
|------|------|----------|------|
| **Phase 1（本周）** | P0: 拆分 DecisionAgent (1413行→4模块)<br>P2: GateController 独立 | 2-3天 | 可维护性↑, 测试覆盖↑ |
| **Phase 2（本月）** | P1: Fan-out 批量评估 + QuoteService 共享行情<br>P4: Supervisor 升级 | 5-7天 | 性能↑ (50%重复调用消除)<br>灵活性↑ (条件分支) |
| **Phase 3（下月）** | P3: Expert Pool 统一记忆<br>闭环追踪 → WeightExpert | 3-5天 | 记忆闭环↑<br>分析可复现↑ |

### 4.3 不值得做的事（过度工程警告）

1. **❌ 完全采用 Hierarchical Delegation**
   - 天枢只有 2 层深度（主流程→Agent），不够资格用递归分解
   - 引入会增加复杂度而不带来价值

2. **❌ 通用的 Agent 通信总线 / 消息队列**
   - 当前文件系统通信（写 JSON → 读 JSON）足够可靠
   - 引入 RabbitMQ / Redis pub-sub 是杀鸡用牛刀

3. **❌ 每个 Agent 跑独立进程 / 容器**
   - 所有 Agent 共享 Python 环境，独立进程增加运维成本
   - 除非单个 Agent 内存>200MB 或需独立扩缩容，否则不值得

4. **❌ Schema 驱动的 Chain-of-Thought 验证**
   - 当前已有 `ScreenOutput`/`ReviewResult`/`DecisionResult` 等 Pydantic schema
   - 再做一层 schema 验证 + 自动重试的通用框架 = 过度工程

5. **⚠️ 谨慎：双盲机制的过度强化**
   - ReviewAgent 声明"双盲"但实际参考快筛驱动级别（行77: `S级驱动可额外+5分`）
   - 要么彻底双盲（不注入驱动级别），要么承认是参考审查
   - 但改造成本 > 收益（当前是合理设计决策）

### 4.4 最终结论

天枢权衡当前架构的**最大问题是 DecisionAgent 职责过载**（1413行，6个职责），其次是**行情重复拉取**（每个Agent独立拉取），再次是**记忆碎片化**（4个孤岛）。

核心改进路径：**Pipeline + Producer-Reviewer** 的 Harness 模式组合已经天然匹配天枢的设计；不需要推翻重来，只需要 **在现有 Pipeline 上增加 Fan-out 并行能力和 Gate 服务化**。

建议用最少的代码改动（<1000行）获得最大的架构收益：**P0（决策拆分）+ P2（Gate独立）** 就能解决80%的问题。
