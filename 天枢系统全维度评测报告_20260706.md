# 天枢系统 全维度深度评测报告

> 评测基准：44个Agent模块，27,677行Python，6个测试文件
> 评测方法：三路并行审计（供应链完整性/评分算法/代码质量）+ 手动深度追溯
> 评测日期：2026-07-06

## 一、全链路运行流程

### 1.1 每日全流程拓扑

```
main.py full_cycle ─┬─ Phase 1: Orchestrator ── 五池状态读取 → 是否执行
                    ├─ Phase 2: News ────────── 新闻抓取+LLM分析
                    ├─ Phase 3: Screen ──────── LLM快筛 → 候选池
                    ├─ Phase 4: Review ──────── LLM审查+过热检测 → 升级
                    ├─ Phase 5: Skeptic ─────── 二审制Gate → 高风险拦截
                    ├─ Phase 6: Decision ────── 决策+写S池 → 兜底引擎
                    ├─ Phase 7: Market ──────── 行情刷新
                    ├─ Phase 8: Pool Refresh ── 四池价格刷新+降级扫描
                    └─ Phase 9: T+1 Tracker ── 闭环追踪
```

### 1.2 池间流转

```
快筛候选池 ──(LLM评分≥75)──→ 重点观察池 ──(决策主推)──→ S级操作池
     ↑                            │                      │
     │                            │ (T+1过期)            │
     │                            ↓                      ↓
     └──── 回流 ←─── 重点观察池 ←── 等比衰减×0.85 ←───┐
                                                        │
                             持仓池 ←──(买入执行)──┘    │
                               │                       │
                          (止损触发)                    │
                               ↓                       │
                            边缘池 ←──(评分<65降级)────┘
                               │
                          (45天过期)
                               ↓
                        ⚠️ 直接蒸发（无回流机制）
```

## 二、业务流程合理性评测

### 🔴 P0-1 S池写入决策链条断裂（已修复）
见上一轮详细分析。`_check_price_position` 因API无MA字段导致所有高位强势股被拦截，S池连续25天空仓。

### 🔴 P0-2 兜底引擎覆盖LLM空仓判断
**发现**：`decision_agent.py L578-684` 中LLM返回"暂无/空仓/等待/观望/不建议"时触发兜底引擎，用模板强行生成买入方案。

**核心风险**：
1. `等待/观望`关键词过于宽泛 → 可能匹配LLM正常文本（如"等待市场确认时机"）
2. **震荡市场**落入 `else` 分支 → 与偏多市场共享相同买入模板，不应激进
3. LLM二次尝试仍拒绝 → 无限递归至模板兜底，无上限保护
4. 兜底模板仓位固定10%（偏多市场建议20-30%时太保守，震荡市建议10-20%时尚可但未区分）

### 🔴 P0-3 准确率计算有误导性缺陷
**发现**：审计确认6处问题：
1. `verify_accuracy.py` 不统计空仓日到分母 → 100%准确率可在推荐少但全对时出现
2. 回头看空仓主导期归零 → 混淆"决策频率"与"准确率"
3. 模拟持有收益使用收盘价 → 不含滑点/成交限制，不反映真实交易
4. **生存者偏差**：退市/停牌股票无行情数据被跳过，只评估存活股票的准确率
5. 降级准确率用60分标准与LLM定义的55-64降级区间不一致
6. "升级准确率"与"实战准确率"样本重叠，非独立指标

### 🟡 P1-1 池间流转存在3处死胡同
**发现**：
1. **边缘池45天过期→直接蒸发**：无回流到候选池的机制，评分回升的标的再无机会进入审查链
2. **快筛候选池14天淘汰→直接蒸发**：同样无回流兜底
3. **S池T+1差标跳过回流→直接蒸发**（本轮修复引入的新路径）：止损触发/亏损>5%的标的不回流重点观察池，但也未回流边缘池

### 🟡 P1-2 scripts/pool_price_refresh.py 绕过统一路径
**发现**：`scripts/pool_price_refresh.py L76-117` 手动实现S池行情刷新，不调用 `pool_manager.refresh_s_operation_prices()`，**绕过降级扫描和评分衰减**。

## 三、核心算法逻辑严谨性评测

### 🔴 P0-4 升级阈值语义漂移

| 位置 | 阈值 | 文件 |
|------|:----:|------|
| thresholds.py `S_POOL_MIN_SCORE` | **75** | 官方定义 |
| CLAUDE.md 声明 | ≥75 | 文档 |
| gate_controller.py S级准入 | **≥80** | L220（漂移!） |
| decision_agent.py 决策候选 | ≥75 | L583 |
| track_recorder.py 可操作 | **≥70** | L65（漂移!） |
| review_agent.py 升级判断 | ≥75 | L1025 |
| pool_manager.py A级判定 | **≥70** | L1008（漂移!） |

`gate_controller.py` 使用 **≥80** 而非官方 ≥75，75-79分的标的永远无法进入S池。

### 🔴 P0-5 60-64分响应盲区

```
LLM提示词:  55-64分=降级  65-74分=保留  ≥75分=升级
代码阈值:   <60强制降级    <65存量扫描      ≥75升级
              ↑               ↑
           60-64盲区 ←──── 代码不触发降级，
                             但LLM提示词说应降级
```

`HARD_DOWNGRADE_SCORE=60` 与 LLM提示词定义的降级区间 **55-64** 不匹配。60-64分的标的依赖LLM自觉触发降级，代码的自动降级引擎不会处理。`review_agent.py L452/467` 只检查 `<60`。

### 🟡 P1-3 评分零值扩散路径

审计确认4条路径可致评分=0：
1. CRITICAL过热扣30分 → `max(0, score-30)` 极低分标的归零
2. WARNING过热扣分 → 3分标的扣5分后=0
3. 一票否决强制降级 → `max(0, min(score,54))` 已为0的score保持0
4. LLM正则解析默认值 → `'综合评分[^\\d]*?(\\d+)'` 匹配失败时score=0

**核心风险**：第一次 `_apply_pool_updates` 解析得到score=0后不会被市场通缩 `max(40,0)` 救援——市场通缩只在第二次解析后执行，但池更新已基于第一次的0分执行。

### 🟡 P1-4 过热检测未覆盖盘中价格刷新

`OverheatDetector.detect()` 只在 `review_agent.py`（审查阶段L1258）和 `quality_gate.py`（S池准入L169）被调用。**4个refresh函数均不触发**：

```
refresh_key_watch_prices()       ❌ 无过热检测
refresh_holdings_prices()        ❌ 无过热检测
refresh_screen_candidate_prices()❌ 无过热检测
refresh_s_operation_prices()     ❌ 无过热检测
```

盘中价格异动时错过过热拦截机会。

### 🟡 P1-5 市场状态通缩无安全余量

震荡市通缩3分后，DYNAMIC准入阈值78分→75分。但 `decision_agent.py L581` 兜底引擎检查固定使用 `>=75`，不随市场状态调整。偏空市场应提高准入至85分而非固定的75分。

## 四、代码实现质量评测

### 🔴 P0-6 裸文件操作无异常保护

审计确认12处高风险裸文件操作：

| 位置 | 代码 | 风险 |
|------|------|------|
| main.py:118 | `review_file.read_text(encoding="utf-8")` | 文件被删除→主流程崩溃 |
| decision_agent.py:714 | `json.loads(s_pool_path.read_text())` | 池文件损坏→崩溃 |
| decision_agent.py:728 | `json.loads(kw_pool_path.read_text())` | 同上 |
| pool_updater.py:326 | `path.write_text(json.dumps(...))` | 磁盘满→崩溃 |
| pool_manager.py:81 | `json.loads(pool_file.read_text())` | 文件损坏→崩 |
| screen_agent.py:247 | `json.loads(pf.read_text())` | 同上 |
| gate_controller.py:19 | `json.loads(verdict_file.read_text())` | 同上 |
| statsmodels_analysis.py:356/382 | `write_text` / `read_text` | 同上 |

风险已验证：`read_text()` 抛出 `FileNotFoundError` → 整个 `main.py` 流程崩溃，后续所有阶段不执行。

### 🟡 P1-6 硬编码阈值泛滥

审计确认10处硬编码值未引用 `thresholds.py`：

| 位置 | 硬编码值 | 应引用 |
|------|----------|--------|
| pool_manager.py L1022 | `< 65` | `AUTO_DOWNGRADE_SCORE` |
| pool_manager.py L1125-1130 | `7天/0.5分/15上限/40下限` | 4个SCORE_DECAY_*常量 |
| review_scorer.py OverheatDetector | `CRITICAL_GAIN_PCT=12`等 | `thresholds.py OVERHEAT_*` |
| gate_controller.py L220 | `>= 80` | `S_POOL_MIN_SCORE` |
| review_agent.py L1100 | `<= 65 or (70 <= x < 75)` | thresholds等级阈值 |
| decision_agent.py L222 | `max(55, ...)` | `SCORE_DECAY_FLOOR` |
| pool_updater.py L23 | 导入了`S_POOL_MIN_SCORE`但未使用 | 孤立导入 |

**风险**：`review_scorer.py` 的 OverheatDetector 类常量与 `thresholds.py` 的 `OVERHEAT_*` 常量是两份独立定义，修改阈值时可能只改了一处。

### 🟡 P1-7 测试覆盖率极低

73个.py / 27,677行产品代码 → 仅6个测试文件 / 约400行测试代码。覆盖率估算 <5%。

任何阈值修改、流程调整、新增功能无法通过回归测试验证。

### 🟢 P2-1 持仓池总盈亏计算逻辑缺陷

`refresh_holdings_prices()` 中 `总盈亏=sum(历史持仓盈亏额)`——仅包含已平仓盈亏，不包含当前持仓浮盈浮亏。

### 🟢 P2-2 死代码残留

`_classify_t1_verdict()`、`get_t1_summary()` 定义于 `closed_loop_tracker.py` 但零调用。`pool_updater.py` L23 `from agents.thresholds import S_POOL_MIN_SCORE` 导入但未使用。
