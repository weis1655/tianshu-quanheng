# 质疑审查：策略C（跨期对比）设计方案

> 角色：Skeptic（四轮联动第2轮）
> 审查对象：`回头看.py` 策略C增补设计
> 代码基线：1012行（v3状态，含策略A+B）

---

## 🔴 P0：设计缺陷（必须修复）

### P0-1：趋势检测过于单薄——两期对比噪声过大

**问题描述**：`calc_trend` 只对比最近两期数据（本期 vs 上期），单个数据点的波动就会被标记为"趋势"。从现有报告数据看：

- `实际准确率` 波动范围大：5月25日报告 `N/A` → 5月29日报告数据不同 → 6月3日 `38.5%`，单点波动可达 ±10%+
- 行情波动、文件解析局部失败、网络请求超时都会造成单期指标失真

**风险后果**：趋势标记频繁翻转，用户产生"狼来了"疲劳，最终无视趋势章节。趋势对比沦为噪音表格。

**建议**：采用 **三期移动平均** 对比（前3期均值 vs 本期），或者对变化幅度设置阈值（如 <5% 标记为 `➡️ ~`，不显示具体数值）。

**涉及改动**：改动2 → 需重新设计 `calc_trend` 签名和算法。

---

### P0-2：自回归告警与 detect_p0_issues 架构耦合模糊

**问题描述**：设计方案说"在 `detect_p0_issues` 或报告中增加逻辑"，关键架构决策未定。

**当前实况**：`detect_p0_issues` 是一条纯函数（line 421-517），参数明确、无副作用、无文件依赖。如果在这里注入历史状态读取：

```python
# 假想代码
def detect_p0_issues(..., state=None):  # 新增参数，破坏签名
    ...
    # 需要访问 p0_type_counts
```

会导致：
1. 调用方必须提前 `load_state()` → 函数不再自包含
2. 单元测试需要 mock 状态文件 → 测试复杂度上升
3. 当前 `test_verify.py` 中 `from 回头看 import detect_p0_issues` 的直接导入模式需要改

**风险后果**：架构污染——一个不该知道状态管理的函数被强行塞入 I/O 依赖。

**建议**：把顽固问题检测放在 **报告生成阶段**（`generate_report` 中 report 字符串拼接处）单独判断，不修改 `detect_p0_issues` 签名。改动4应该作为独立的后处理步骤，而非寄生在原有函数中。

---

### P0-3：状态文件存储位置硬编码

**问题描述**：设计中 `STATE_FILE = BASE_DIR / '回顾报告' / '.previous_state.json'`。但当前代码有 `OUTPUT_DIR = BASE_DIR / '回顾报告'`，且 `generate_report` 接受 `output_file` 参数允许自定义输出路径。

**当前实况**：line 16-18、line 1024-1025：
```python
BASE_DIR = Path('/home/seven/hermes-data/tianshu-quanheng/data')
OUTPUT_DIR = BASE_DIR / '回顾报告'
# ───
output_file = OUTPUT_DIR / f"{datetime.now().strftime('%Y-%m-%d')}_回头看报告_v3.md"
```

**风险**：如果未来 `OUTPUT_DIR` 改变或用户指定了 `output_file` 参数，状态文件路径会跟报告实际存储路径不一致。状态文件应该跟报告文件 **同级**，用 `OUTPUT_DIR` 而非硬编码拼接。

**建议**：`STATE_FILE` 引用 `OUTPUT_DIR` 而非 `BASE_DIR`，且在 `generate_report` 中作为局部变量动态构造。

## 🟡 P1：重要改进（建议纳入）

### P1-1：avg_return 跨期不可比——未记录样本量

**设计字段**：`avg_return` 被直接存入 history，但 `avg_return` 取决于该期 `perf_total`（验证推荐总数）。某期验证了5只、下一期验证了30只，`avg_return` 的变化可能完全来自样本量差异而非策略变化。

**当前实况**：line 706-708：
```python
actual_accuracy = round(len(perf_profits) / len(perf_total) * 100, 1) if perf_total else 0
avg_return = round(sum(p['change_pct'] for p in perf_total) / len(perf_total), 2) if perf_total else 0
```

**风险**：趋势表显示 avg_return 下降1.2%，但读者不知道是因为总样本从30只降到5只。**指标本身纯度不够**。

**建议**：history 记录中增加 `verified_count` 字段，趋势表中 avg_return 旁边标注样本数。

---

### P1-2：calc_trend 对字典类型（p0_type_counts）的处理未定义

**设计签名**：`def calc_trend(history, key)` — 但 `p0_type_counts` 是 `{类型: 出现次数}` 字典，不是单一数值。对这个 key 调用 `calc_trend` 应返回什么？

- 每个子类型的趋势？→ 需要返回嵌套结构
- 总类型数变化？→ 丢失了顽固问题的粒度
- 某个特定类型（如"降级延迟"）的连续出现次数？→ 跟 `calc_trend` 的单值返回设计矛盾

**建议**：`p0_type_counts` 应该用专门的 `detect_stubborn_issues()` 函数处理，而不是复用 `calc_trend`。`calc_trend` 只处理数值/百分比字段。

---

### P1-3：状态文件损坏的无感静默

**方案描述**：`load_state` 读取失败返回默认值。但如果 `.previous_state.json` 存在但部分损坏（某字段类型错误、JSON语法错、中途写入被截断），当前方案会直接覆盖为默认值，丢失所有历史数据且无人知晓。

**当前实况**：`.price_cache.json` 已有类似风险（line 96-107），且目前没有静默失败的报警。

**建议**：
- 损坏时打印 `[回头看] ⚠️ 状态文件损坏，历史趋势数据将丢失: {e}`
- 写入时使用 **原子写入**（先写临时文件再 rename），防止写入中断导致文件截断

---

### P1-4：save_state 写时机可能导致跨期数据丢失

**当前方案**：在 `generate_report` 末尾调用 `save_state()`。

**风险**：如果 `generate_report` 中途（行情API超时、文件读取IOError、内存异常）崩溃，状态不会保存，但报告文件可能已部分写入。下次运行看到的是空白状态，跨期趋势丢失。

**建议**：在计算完所有指标（line 710 之前）立即 `save_state()`，而不是在报告拼接结束时。但需要注意 `save_state` 中的 `history` 追加需要在 `generate_report` 中逐步构建。

---

### P1-5：history 保留10期 vs 实际只用3期——存储与使用不匹配

**设计方案**：`history` 保留最近10期，但自回归告警只取最近3期。

**问题**：10期的存储规模有无业务依据？如果无明确用途，属于预设未来需求的过度设计。保留10期意味着：
- 状态文件体积随报告版本累积
- history 中过期数据可能被不小心用于未来分析
- `calc_trend` 如果未来改成多期统计（如5期均线），API 签名不兼容

**建议**：要么只留5期（覆盖2周的日/维度，充足覆盖3期分析+2期缓冲），要么明确定义第4-10期的用途（如"趋势加速/减速判断"或"季节性模式分析"）。

## 🔵 P2：锦上添花（可选优化）

### P2-1：趋势表位置建议——避免信息冗余

**设计方案**：趋势表放在 P0预警之后、快筛回顾之前。

**问题**：趋势表中的所有"本期"数据（P0数量、实战准确率、avg_return）都会在报告后面的章节中被详细展示。用户在前言看到一张浓缩表格，然后在之后的4个章节中看到同样的数据被展开。这种重复降低报告密度。

**建议**：将趋势表移到 **第五章「综合评估」末尾、第六章「改进建议」之前**，作为跨期汇总的收尾。或者直接将趋势数据嵌入综合评估的评价体系中。

---

### P2-2：状态文件并发写竞态

**风险**：如果 `--cron` 模式和手动运行同时发生，两个进程同时 `load_state()` → `修改` → `save_state()`，后保存的会覆盖之前的，丢失其中一期的数据。

**建议**：目前可忽略（单人使用+Hermes 单进程的限制），但架构上可用 `filelock` 或通过 `append` 方式（文件名按日期分片）规避。

---

### P2-3：历史报告文件已包含可用数据

**建议**：现有历史报告（如 `2026-06-03_回头看报告_v3.md`）已经包含了所有统计指标。理论上可以通过解析历史报告来重建 history，不需要额外的 `.previous_state.json`。作为可选的长远方向，考虑将历史报告本身作为数据源，减少一次额外的 I/O 写入。

---

### P2-4：趋势 emoji 的终端兼容性

**问题**：`🔴 +13`、`🟡 -3.5%`、`⛔` 等在所有终端/CI 输出中不一定正确渲染。如果报告被输出到 stdout 并被 cron 邮件捕获，emoji 可能变成乱码。

**建议**：增加 `PLAINTEXT_MODE` 回退方案（`[R] +13`、`[Y] -3.5%`、`[STUBBORN]`）。

| 优先级 | 改动项 | 原因 |

## 📊 总结

| 级别 | 数量 | 关键问题 |
|------|------|----------|
| 🔴 P0 | 3 | 两期对比噪声 / detect_p0_issues 架构污染 / 路径硬编码 |
| 🟡 P1 | 5 | 指标可比性 / 类型不匹配 / 静默失败 / 写时机 / 存储过度 |
| 🔵 P2 | 4 | 位置冗余 / 并发竞态 / 历史文件复用 / emoji兼容 |

**核心结论**：策略C的方向正确（跨期对比是成熟的回顾分析手段），但当前设计在 **趋势检测颗粒度**（两期太粗糙）和 **架构嵌入方式**（改动4寄生于现有函数）两个关键点上存在显著风险，建议先修复 P0-1 和 P0-2 后再进入实现阶段。
