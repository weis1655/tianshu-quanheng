## 四、进化路径推荐

### 推荐执行顺序

```
🅰 大函数拆分 → 🅲 核心测试 → 🅱 异常治理 → 🅳 架构加固
```

**理由：**
1. **🅰 拆分** 收益最高风险最低 — 纯重构，不改变业务逻辑，快速改善架构
2. **🅲 测试** 在拆分后立即写 — 锁定拆分后的接口，防止回归
3. **🅱 异常** 依赖前两步稳定 — 有测试保护下改异常处理才安全
4. **🅳 架构** 长期迭代 — 需券商接口和回测引擎，非短期可完成

### 预期进化轨迹

```
基线: 72.5/100
  → 🅰 拆分: 72.5 → 80.5 (+8.0)
  → 🅲 测试: 80.5 → 85.5 (+5.0)
  → 🅱 异常: 85.5 → 91.5 (+6.0)
  → 🅳 架构: 91.5 → 100+ (长期)
```

## 六、结论

**天枢权衡 当前进化评分：72.5/100（🟡 中上）**

**已修复的 7 项缺陷贡献约 +13 分**，核心问题（评分提取、LLM退化、边界溢出）已闭环。

**最大残余风险：** `decision_agent._run_impl` 802 行函数 — 单点故障风险极高，任何修改都可能导致连锁问题。

**建议下一步：** 执行 🅰 大函数拆分策略，将 `_run_impl` 拆分为 8-10 个独立方法，每方法不超过 100 行。
