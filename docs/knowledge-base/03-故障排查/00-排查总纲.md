## 一、概述

本文档是天枢权衡系统故障排查的总纲，覆盖以下内容：
- **故障分类体系**：系统化定义所有已知故障类型与严重级别
- **排查方法论**：系统性排查问题的思维框架与原则
- **通用排查流程**：从故障现象到根因的标准操作流程
- **日志查看命令**：各日志文件位置、用途、常用查询命令
- **工具链**：辅助排查的监控脚本与诊断工具

适用于运维人员、系统开发者及故障响应者快速定位并修复问题。

## 二、故障分类体系

### 2.1 严重级别定义

| 级别 | 定义 | 响应时限 | 示例 |
|:----:|------|:--------:|------|
| **P0** | 系统核心功能中断或数据丢失，产生实际资金风险 | 即时 | 降级延迟、错误买入、池数据损坏 |
| **P1** | 次要功能不可用，但核心流程可手动补跑 | 2小时内 | 邮件投递失败、Cron单次跳过 |
| **P2** | 功能异常但可自动恢复，不影响当日运行 | 当日 | 格式漂移、池容量超限、非关键模块报错 |
| **P3** | 偶发问题或低影响异常，无实际业务影响 | 本周 | 日志格式异常、编辑器警告、冗余代码 |

### 2.2 故障分类

根据历史故障记录，天枢权衡系统的故障分为以下11类：

| 故障ID | 分类名称 | 典型现象 | 最高级别 |
|--------|---------|---------|:--------:|
| TROUBLE-01 | 排查总纲 | — | — |
| TROUBLE-02 | 新闻拉取失败 | 新闻源不可用、超时、代理问题 | P1 |
| TROUBLE-03 | LLM调用失败 | API超时、模型拒绝、退化检测 | P0 |
| TROUBLE-04 | 格式漂移 | LLM输出格式变化导致正则提取失败 | P0 |
| TROUBLE-05 | 降级延迟 | 评分0未降级、低分滞留池中 | P0 |
| TROUBLE-06 | 池异常 | 跨池重复、空池、容量溢出、字段缺失 | P0 |
| TROUBLE-07 | Cron任务异常 | 串行阻塞、超时截断、SIGTERM杀进程 | P1 |
| TROUBLE-08 | 决策异常 | 空仓误报、兜底引擎误触、S池残留 | P0 |
| TROUBLE-09 | 邮件投递失败 | 反垃圾拦截、脚本报错、附件问题 | P1 |
| TROUBLE-10 | 回头看异常 | 口径偏差、数据提取失败、重复计数 | P1 |
| TROUBLE-11 | 回测异常 | 双截断、随机因子、重复计数 | P2 |
| TROUBLE-99 | 历史故障案例库 | 所有P0/P1故障的根因+修复+验证 | — |

### 2.3 故障类型矩阵

每条故障可从以下维度分类：

| 维度 | 分类 | 说明 |
|------|------|------|
| **根因类型** | 流程控制缺失 | 某环节应该触发但没触发，修代码 |
| | 阈值不准 | 边界值不合理/条件过严过宽，修数值 |
| | 外部依赖失效 | LLM API、新闻源、行情数据源不可用 |
| **影响范围** | 全流程阻断 | 整个流水线中断 |
| | 单阶段失败 | 某一Agent输出异常，后续可降级运行 |
| | 数据质量下降 | 流程正常但数据不准 |
| **恢复方式** | 自动恢复 | 重试/降级后自行恢复 |
| | 手动补跑 | 需人工执行补跑命令 |
| | 代码修复 | 需修改代码后重新部署 |

## 三、排查方法论

### 3.1 核心原则

1. **先复现，后分析**：不确认现象就盲目翻代码，效率极低
2. **从日志出发**：日志是唯一的权威来源，代码是辅助理解
3. **由外到内**：先检查外部依赖（API/网络/数据源），再查内部逻辑
4. **最小改动**：修复时只改1-2行，不重构（详见 `CLAUDE.md` 修复原则）
5. **验证闭环**：修复后必须确认故障不再复现，且未引入新问题

### 3.2 两类问题区分法

这是盟主指定的核心方法论（2026-06-18）：

| 问题类型 | 特征 | 修复方案 | 实案例 |
|---------|------|---------|--------|
| **流程控制缺失** | 某环节应该触发但没触发/时序错误/条件遗漏 | 加流程守卫/调执行顺序/补过滤条件 | 候选池未过滤滞留股、评分调整后无降级检查 |
| **阈值不准** | 边界值不合理/条件过严过宽 | 调数值/改比较符号 | 系统评分阈值经多轮校准不建议随意动 |

> **重要**：先判断问题是流程缺失还是阈值不准。流程缺失修代码，阈值不准需要先问盟主。

### 3.3 四阶段排查法

```
Phase 1: 现象确认
  └─ 用户报告什么？日志环境复现了吗？
  └─ 最近一次正常运行的版本/时间点

Phase 2: 定界
  └─ 是单次故障还是持续故障？
  └─ 影响范围：全流程阻断 / 单阶段 / 单个标的
  └─ 外部依赖（API/网络/数据源）是否正常？

Phase 3: 根因定位
  └─ 查日志（scheduler / agent / prompts）
  └─ 查代码（函数调用链 + 变量值追踪）
  └─ 查数据（池JSON / 决策日志 / 报告文件）

Phase 4: 修复与验证
  └─ 最小改动修复
  └─ 编译验证：python -m py_compile <file>
  └─ 功能验证：运行对应阶段
  └─ 回归验证：确认未引入新问题
```

### 3.4 常见排查陷阱

| 陷阱 | 说明 | 避免方法 |
|------|------|----------|
| **只看 last_run_at** | `cronjob list` 的 last_run_at 只能告诉上次跑了没有，不能告诉怎么死的 | 权威来源是 `agent.log` 的 scheduler 日志 |
| **LLM推荐 vs 系统降级矛盾** | 先查日志确认降级是LLM打分还是硬规则/衰减导致 | 查 `logs/YYYY-MM-DD.log` 搜索"降级" |
| **100%评分提取失败** | 所有标的同时失败 → 格式漂移；单个标的持续失败 → 正则覆盖不全 | 对比 `logs/prompts/` 中原始输出与正则模式 |
| **SIGTERM杀进程误判为Cron未触发** | 有 Running job 但无 completed → 被杀了 | 查 `agent.log` 中 SIGTERM 信号 |

## 四、通用排查流程

### 4.1 快速诊断清单

当接到故障报告时，按以下顺序快速排查：

```
Step 1: 确认系统状态
  python main.py status          → 五池状态（池是否为空、容量是否超限）
  python main.py health           → 健康检查（所有模块可导入）
  python main.py circuit          → 熔断器状态（API是否被熔断）

Step 2: 检查Cron任务（如涉及定时任务）
  grep "cron.scheduler" ~/.hermes/logs/agent.log
  → 查 Running job / completed successfully / delivered

Step 3: 检查今日日志
  tail -100 logs/$(date +%Y-%m-%d).log
  → 查 ERROR / WARNING / 异常堆栈

Step 4: 检查LLM调用日志
  ls -lt logs/prompts/ | head -5
  → 查看最新LLM prompt/output，确认格式是否正常

Step 5: 检查数据完整性
  python scripts/data_quality_monitor.py
  → 五池数据质量扫描（空池/重复/字段缺失）
```

### 4.2 各阶段故障诊断入口

| 故障现象 | 第一步排查 | 日志文件 | 参考文档 |
|---------|-----------|---------|---------|
| 新闻未拉取 | 检查新闻源可用性 | `logs/YYYY-MM-DD.log` 搜索"news" | TROUBLE-02 |
| 快筛无输出 | 检查LLM调用是否成功 | `logs/prompts/ScreenAgent_*.md` | TROUBLE-03 |
| 审查评分异常 | 检查V2正则提取是否成功 | `logs/prompts/ReviewAgent_*.md` | TROUBLE-04 |
| 低分未降级 | 检查评分衰减 + 降级扫描 | `logs/YYYY-MM-DD.log` 搜索"降级" | TROUBLE-05 |
| 池重复/空池 | 运行数据质量监控 | `五池管理/*.json` 手动检查 | TROUBLE-06 |
| Cron未执行 | 查 scheduler 日志 | `~/.hermes/logs/agent.log` | TROUBLE-07 |
| 决策报告异常 | 查LLM退化计数器 | `data/llm_degradation_counter.json` | TROUBLE-08 |
| 邮件未收到 | 查邮件脚本退出码 | `logs/email_failures.log` | TROUBLE-09 |
| 回头看数据不准 | 查统计口径 | `data/decision_log.json` | TROUBLE-10 |
| 回测结果异常 | 查随机因子/截断 | `data/auto_tasks/backtest.log` | TROUBLE-11 |

### 4.3 补跑操作

当故障修复后，需要补跑受影响阶段：

```bash
# 全流程补跑
cd ~/hermes-data/tianshu-quanheng
python main.py full

# 单阶段补跑
python main.py news           # 新闻分析
python main.py screen         # 快筛
python main.py review         # 审查
python main.py decision       # 决策
python main.py ts             # 时间序列分析

# 运维操作
python main.py status         # 五池状态
python main.py weekly         # 周复盘
python main.py health         # 健康检查
python main.py metrics        # 运行指标
python main.py circuit        # 熔断器状态
python main.py feedback       # 反馈闭环
```

### 4.4 升级与上报

当故障满足以下条件时，需升级处理：

- **P0**：即时响应，修复后1个工作日内写入 TROUBLE-99 历史故障案例库
- **连续3期同类P0**：升级为系统性工单式整改（详见 `SKILL.md` 工单式修复方法论）
- **外部依赖故障**：确认不是自身代码问题后，上报给对应服务商

| 结构化日志 | `logs/tianshu_structured.jsonl` | JSON格式的结构化日志 | JSON Lines |

### 5.2 常用查询命令

```bash
# 查今日运行日志
tail -100 logs/$(date +%Y-%m-%d).log

# 查指定日期日志
grep "ERROR\|WARNING" logs/2026-07-24.log

# 查特定Agent日志
grep "ReviewAgent\|DecisionAgent" logs/2026-07-24.log

# 查降级相关日志
grep "降级\|downgrade\|边缘池" logs/2026-07-24.log

# 查评分相关日志
grep "评分\|score\|综合分" logs/2026-07-24.log

# 查异常堆栈
grep "Traceback\|Error\|Exception" logs/2026-07-24.log

# 查Cron调度日志
grep "cron.scheduler" ~/.hermes/logs/agent.log | tail -20

# 查Cron任务执行详情
grep "cron.scheduler.*盘中复盘\|cron.*盘中复盘" ~/.hermes/logs/agent.log

# 查SIGTERM信号
grep "SIGTERM\|shutdown" ~/.hermes/logs/agent.log

# 查Gateway启动状态
grep "Starting Hermes Gateway\|Cron ticker started" ~/.hermes/logs/agent.log | tail -2

# 查看最新LLM Prompt
ls -lt logs/prompts/ | head -5
cat logs/prompts/$(ls -t logs/prompts/ | head -1)

# 查LLM退化状态
cat data/llm_degradation_counter.json

# 查邮件发送失败
cat logs/email_failures.log

# 查合规日志
cat data/compliance/compliance_$(date +%Y-%m-%d).jsonl
```

### 5.3 日志分析技巧

**判断是否被SIGTERM杀进程**：
```bash
# 有 Running job 但无 completed successfully → 被杀了
grep "cron.scheduler: Running job" ~/.hermes/logs/agent.log | grep "10:3[0-9]"
grep "cron.scheduler: completed successfully" ~/.hermes/logs/agent.log | grep "10:3[0-9]"
```

**判断串行阻塞**：
```bash
# 检查前一个任务是否还在跑导致后续任务延迟启动
grep "cron.scheduler: Running job" ~/.hermes/logs/agent.log | grep -E "06:1[5-9]|06:2[0-9]|06:3[0-9]"
```

**判断模型安全拒绝**：
```bash
# 查Cron输出目录的错误日志
ls ~/.hermes/cron/output/<job_id>/
cat ~/.hermes/cron/output/<job_id>/$(ls -t ~/.hermes/cron/output/<job_id>/ | head -1)
```

**判断格式漂移**：
```bash
# 对比LLM原始输出与正则模式
cat logs/prompts/ReviewAgent_$(date +%Y%m%d)*.md | head -100
# 搜索 review_agent.py 中 V2 正则模式
grep -A5 "V2_PATTERNS\|RE_V2_" agents/review_agent.py
```

## 七、故障响应流程

### 7.1 P0故障响应

```
1. 发现故障 ──→ 2. 立即止损 ──→ 3. 根因定位 ──→ 4. 最小修复 ──→ 5. 验证 ──→ 6. 记录案例
   (用户报告/    (停止问题     (参考本总纲    (参考CLAUDE.md  (编译验证+    (追加到
    自动检测)     继续执行)     排查流程)      修复原则)      功能验证)    TROUBLE-99)
```

### 7.2 P1故障响应

```
1. 确认故障 ──→ 2. 评估影响 ──→ 3. 手动补跑 ──→ 4. 排期修复 ──→ 5. 记录案例
   (日志确认)    (是否影响    (python main.py   (当日/次日    (TROUBLE-99)
                 核心流程)     full)            修复)
```

### 7.3 故障复盘

每次P0/P1故障修复后，需在5个工作日内完成复盘：

- 根因分析是否正确？
- 修复是否完全（无类似问题残留）？
- 排查流程是否可优化？
- 文档是否需要更新？
