# 天枢权衡定时任务管理

## 1. 概述

天枢权衡系统依赖 Hermes Agent 内置的 Cron Scheduler 驱动每日自动运行。所有定时任务均注册在 `~/.hermes/cron/jobs.json` 中，由 Hermes Gateway 的 scheduler 组件负责调度执行。

本文档覆盖天枢系统所有 Cron 任务的时间线、调度规则、异常处理、补跑策略及常见问题排查。

### 2.3 依赖链说明

1. **07:10 早盘全流程**（fundamental_cycle）：独立执行 `python main.py full`，跑完新闻→快筛→审查→SkepticGate→决策五阶段。不依赖其他 Cron 任务。
2. **10:30 盘中复盘**：依赖早盘生成的决策报告、审查报告、五池 JSON 文件。若早盘任务因超时/失败未生成决策报告，盘中复盘读取时需 try/except 兜底。
3. **14:30 临盘决策**：依赖盘中已更新的五池数据文件。若早盘全流程 14:30 前未完成（极端情况），临盘决策可能读取到过期池数据。
4. **15:00-15:30 收盘后任务链**：回头看(15:00)→反馈闭环(15:06)→运营日报(15:10)→自动迭代(15:30)。此链存在串行阻塞风险（见第5节）。

## 3. 调度窗口机制

### 3.1 时间窗口规则

`python main.py` 根据运行时间自动选择执行模式（来自 `main.py` 调度逻辑）：

| 时间窗口 | 默认执行 | 说明 |
|---------|---------|------|
| 06:15-06:35 | **news_only**（仅新闻） | 7:00前仅执行新闻分析，不触发交易阶段 |
| 07:05-07:20 | **full_cycle**（全流程） | 早盘合并窗口，自动执行新闻→快筛→审查→SkepticGate→决策 |
| 其他时间 | **full_cycle**（全流程） | 非窗口时间默认全量执行 |

> ⚠️ **窗口强制全流程**：在 06:15-06:35 窗口运行 `python main.py` 默认只跑新闻。如需在此窗口强制全流程，需显式传参：`python main.py full`。

### 3.2 窗口与 Cron 任务对齐

实际 Cron 任务时间与调度窗口的对应关系：

| Cron任务 | 调度时间 | 所在窗口 | 备注 |
|---------|---------|---------|------|
| 早盘四段闭环 | 07:10 | ✅ 07:05-07:20 full_cycle窗口 | 完全对齐，自动full_cycle |
| 盘中复盘 | 10:30 | 非窗口 | 传参 `full` 强制全流程 |
| 临盘决策 | 14:30 | 非窗口 | 传参 `full` 强制全流程 |

## 4. 周末模式

### 4.1 行为规则

周末（周六/周日）执行 `python main.py` 自动进入周末模式：

- ✅ **新闻分析正常执行**：宏观前置分析照常运行
- ✅ **池健康检查**：清理超14天陈旧标的
- ❌ **跳过所有交易阶段**：快筛/审查/SkepticGate/决策/复盘
- ❌ **不生成技术面分析/审查报告/决策报告**：交易相关文件不落地
- 📧 **邮件正常发送**：交易文件区域标注"不存在"
- 📝 **报告聚焦**：新闻要点 + 地缘风险 + 下周一展望

### 4.2 周末Cron任务行为

| Cron任务 | 周末行为 | 说明 |
|---------|---------|------|
| 🦎 新闻联播分析（06:20） | ✅ 正常执行 | 新闻分析全日无休 |
| ⚖️ 早盘四段闭环（07:10） | ⚠️ 进入周末模式 | 跑新闻+池检查，跳过交易阶段 |
| ⚖️ 盘中复盘（10:30） | ⚠️ 进入周末模式 | 同上 |
| ⚖️ 临盘决策（14:30） | ⚠️ 进入周末模式 | 同上 |
| ⚖️ 反馈闭环（15:06） | ❌ 不执行 | `expr: 6 15 * * 1-5`，仅工作日 |
| ⚖️ 周复盘（周六09:00） | ⚠️ 特殊处理 | 见4.3节 |
| ⚡ 池价格刷新 | ❌ 不执行 | `expr: */15 9-14 * * 1-5`，仅工作日 |

### 4.3 周复盘周六 Fallback

`python main.py weekly` 在周六会进入周末模式（`weekly_review_agent.py weekday()>=5` 守卫），返回结果：

由于无报告文件落地，周六 09:00 的周复盘 Cron 邮件阶段需自行组合报告，数据来源：

## 5. Cron串行阻塞问题

### 5.1 问题描述

**Hermes Cron Scheduler 是单线程串行执行**。一个长任务会阻塞后续所有已排期的任务，直到前一个任务完成或超时。

### 5.2 典型阻塞场景

```
06:20 🦎 新闻联播分析（长任务，30+分钟）...
07:10 ⚖️ 早盘四段闭环（被阻塞，实际07:XX才启动）...
10:00 🏛️ 新闻联播分析（被阻塞，实际10:XX才启动）...
10:30 ⚖️ 盘中复盘（被阻塞，实际10:4X才启动）
       ↓
10:40:17 gateway收到 systemd SIGTERM → 进程被杀死
       ↓
新 gateway 启动，排期已过 → 已错过的任务不会重跑
```

### 5.3 诊断方法

```bash
# 查 scheduler 日志确认任务启动时间
grep "cron.scheduler: Running job" ~/.hermes/logs/agent.log | grep "10:2[0-9]\|10:3[0-9]"

# 查"有启动无完成"——任务被杀了
grep "cron.scheduler.*盘中复盘" ~/.hermes/logs/agent.log
# 如果只有 Running job 但没有 completed successfully，说明任务被中断

# 查 SIGTERM 信号
grep "SIGTERM\|shutdown" ~/.hermes/logs/agent.log | grep -E "10:4[0-5]"
# 输出示例：Received SIGTERM — initiating shutdown
```

### 5.4 缓解措施

- **长任务提前调度**：新闻联播分析（06:20）已安排在尽可能早的时间窗口
- **拆分任务**：反馈闭环(15:06) 与 自动迭代(15:30) 之间保留24分钟缓冲
- **手动补跑**：被阻塞/被杀的任务需手动补跑（见第7节）
- **注意**：新 gateway 启动后已错过的任务不会自动重跑，这是 Hermes 基础设施特性

## 6. 超时处理

### 6.1 Full Cycle 120s 超时截断

`python main.py full`（full_cycle）在 120s terminal timeout 内可能被 SIGTERM 杀进程。2026-06-23 实测各阶段耗时：

| 阶段 | 耗时 | LLM调用次数 | 是否完成（120s内） |
|------|:----:|:----------:|:----------------:|
| NewsAgent | ~17s | 1次 | ✅ |
| MarketAgent | ~3s | 0次 | ✅ |
| ScreenAgent | ~11s | 2次 | ✅ |
| ReviewAgent | ~89s | 1次 | ✅（临界完成） |
| DecisionAgent | 未触发 | 1次 | ❌ SIGTERM 115s |

**总耗时约 134s 才能完成全五阶段，超出 120s 默认终端超时。**

### 6.2 检测方法

```bash
# 检查今日决策报告是否生成
ls -la data/历史记录/*_决策报告.md
# 或用 mtime 过滤最近生成的
find data/历史记录/ -name "*决策报告*" -mmin -5
```

### 6.3 恢复步骤

```bash
# Step 1: 单独执行 decision 阶段（通常 10-15s）
cd ~/hermes-data/tianshu-quanheng
python main.py decision

# Step 2: 验证决策报告内容
cat data/历史记录/$(date +%Y-%m-%d)_决策报告.md | head -20

# Step 3: 发送邮件（如尚未发送）
python3 scripts/tianshu_email_template.py --type 临盘决策 --to sjj139@139.com
```

### 6.4 根本原因与替代方案

**根因**：120s 是 Hermes terminal() 工具的默认超时。review 阶段是最长的单一阶段（~89s），因为除了 LLM 审查外还包含过热检测（WARNING-1/2/3/4）、池更新（升级/降级）、边缘池溢出清理、价格刷新等后处理。

**替代方案**：
- 拆分为两阶段执行：`python main.py full`（不跑 decision） + `python main.py decision`
- 或在 Cron prompt 中增加 timeout 参数（180s+）
- 或在执行后检测超时，自动补跑 decision 阶段

## 8. 模型安全拒绝

### 8.1 问题描述

`deepseek-v4-flash`（通过 SenseNova API 调用）在股票分析场景时间歇触发安全策略，拒绝生成汇总报告。典型表现为：

- Cron 状态显示 `ok`，但最终输出是「你好，我无法给到相关内容」
- 任务本身已跑完 `main.py full`，报告文件都在，但 LLM 最后的汇总输出被拦截
- 间歇性发生，无明显规律

### 8.2 修复历程

| 日期 | 操作 | 结果 |
|------|------|------|
| 2026-06-08 | 换 `anthropic/claude-sonnet-4-20250514` + `opencode-zen` | ❌ 失败——opencode-zen 不支持此模型 |
| 2026-06-09 | 恢复 `deepseek-v4-flash` + `custom`（商汤 SenseNova） | ✅ 确认可用 |

### 8.3 重试与Fallback机制

天枢系统在 `_call_llm_sensenova()` 中实现了重试+fallback：

1. **主模型重试**：`deepseek-v4-flash` 失败后自动重试3次
2. **降级fallback**（2026-06-25 新增）：3次全部失败后，自动降级至 `sensenova-6.7-flash-lite`（即使主模型是 deepseek-v4-flash）
3. **Cron Prompt 免责声明**：所有天枢 Cron 任务的 prompt 前缀均包含 `[IMPORTANT DISCLAIMER]` 声明，降低安全策略触发概率

### 8.4 诊断方法

```bash
# 查看 cron 输出目录的错误日志
ls ~/.hermes/cron/output/<job_id>/
cat ~/.hermes/cron/output/<job_id>/$(ls -t ~/.hermes/cron/output/<job_id>/ | head -1)

# 查 provider 支持的模型
cat ~/.hermes/provider_models_cache.json | python3 -m json.tool

# 直接测试 API
curl -s -X POST https://opencode.ai/zen/v1/chat/completions \
  -H "Authorization: Bearer $OPENCODE_ZEN_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"deepseek-v4-flash-free","messages":[{"role":"user","content":"hi"}],"max_tokens":20}'

# 更新 cron 模型
# cronjob action=update job_id=xxx model='{"model":"deepseek-v4-flash","provider":"custom:deepseek-v4-flash-sensenova"}'
```

## 9. 后执行报告编译模式

### 9.1 概述

Cron 任务执行 `main.py` 完毕后，需按既定链路读取产出文件 → 提取关键数据 → 编译为纯文本投递报告（飞书/139邮箱均不渲染 Markdown 表格）。

### 9.2 四类Cron的取数差异

| Cron 类型 | 必读文件 | 可选文件 |
|-----------|---------|---------|
| 早盘全流程 (07:10) | `*_快筛报告.md` + `*_审查报告.md` + `*_决策报告.md` + 五池 JSON | `*_宏观前置分析.md` |
| 盘中复盘 (10:30) | `*_技术面分析.md` + `*_审查报告.md` + `*_决策报告.md` + 五池 JSON | `*_快筛报告.md` + `*_质疑审查报告.md` |
| 临盘决策 (14:30) | `*_决策报告.md` + S级操作池.json + 持仓池.json | `*_审查报告.md` |
| 反馈闭环 (15:06后) | 闭环追踪 JSON + 持仓池.json + 胜率统计 | `*_审查报告.md` |
| 周末模式 | `*_宏观前置分析.md` + 五池 JSON | 无（交易文件标注"不存在"） |

### 9.3 格式约束

## 10. 常见问题速查表

| 症状 | 根因 | 修法 |
|------|------|------|
| 有 Running job 日志，无 completed 日志 | SIGTERM 杀进程/超时 | 手动补跑 `python main.py decision` |
| 启动时间比预定晚 X 分钟 | 前一个长任务阻塞 | 调整调度时间或拆分任务 |
| 无 Running job 日志 | 排期未触发 / gateway 未启动 | 检查 gateway 状态 |
| 有 completed 但无 delivered | 投递失败（SDK 断开/代理不可达） | 检查飞书/139邮箱连接 |
| 状态=ok 但输出为空/拒绝文本 | 模型安全策略拦截 | 重试或换 provider |
| 只生成 news 阶段，无交易阶段 | 在06:15-06:35窗口运行 | 传 `full` 参数强制全流程 |
| 邮件报 `ModuleNotFoundError: path_config` | 邮件脚本导入路径问题 | 直接用 `send_email` fallback 直发 |
| 周末跑完无报告文件 | 周末模式正常行为 | 从 pool status + 决策日志自行组合报告 |
| 交易阶段全跳过 | 周末模式 | 正常行为，无需处理 |
